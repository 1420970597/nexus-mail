# 极径 | Nexus-Mail

**Nexus-Mail** 是一个面向自动化场景的邮件接码与验证码提取平台，产品逻辑参考成熟的 **SIM 接码平台**（如 5SIM、SMS-Activate、SMS-Man 一类平台）的通用模式来设计，但把“手机号 / SIM 卡 / 短信”替换为“邮箱地址 / 域名资源 / 邮件”。

它的核心目标不是做传统邮箱服务，而是做一个 **可编排、可计费、可审计、可供应商化** 的验证码邮件接收中台：

> **文档定位**：当前 README 是面向产品定义与系统设计的草案，重点回答“这个平台应该如何按照 SIM 接码平台的逻辑来设计”，并不代表所有接口与状态字段已经最终定稿。

- 面向 **用户** 提供按次购买、轮询查询、Webhook 回调、再次取码、历史订单查询等能力
- 面向 **供应商** 提供域名 / 邮箱池 / 转发规则 / 提取规则 / 库存与成本管理能力
- 面向 **管理员** 提供定价、风控、审核、资金、争议处理、供应商管理和平台运营能力

---

## 1. 设计出发点：把 SIM 接码逻辑映射到邮箱接码

成熟 SIM 接码平台的共性并不在“短信”本身，而在以下业务抽象：

1. **资源池**：号码池 / 库存池
2. **按项目分配**：Telegram、Google、TikTok 等项目对应不同价格、成功率、库存
3. **按次购买**：用户先充值，再购买一次激活会话
4. **订单生命周期**：等待资源 → 等待验证码 → 收到验证码 → 完成 / 超时 / 取消 / 拉黑
5. **状态驱动计费**：未收到码可退款，收到码则扣费成立
6. **供应商体系**：平台不一定自己生产全部资源，往往接入多个上游或码商
7. **风控与评分**：防止恶意取消、薅退款、批量滥用、垃圾请求
8. **自动化 API**：余额、价格、库存、下单、轮询结果、回调通知、订单历史

Nexus-Mail 采用同样的结构，只是把资源对象换成：

- **SIM 平台中的号码** → **Nexus-Mail 中的邮箱地址 / 别名地址**
- **国家 / 运营商** → **域名 / 邮箱池 / 来源供应商 / 路由节点**
- **短信内容** → **邮件主题 / 发件人 / HTML / Text / 附件 / 提取结果**
- **短信验证码** → **邮件验证码 / Magic Link / Reset Link / OTP / 邀请链接**

换句话说，Nexus-Mail 本质上是：

> 一个遵循接码平台订单逻辑的“邮件接码交易与分发平台”。

## 2. 参考的现有 SIM 接码平台通用逻辑

基于对 5SIM、SMS-Activate / 兼容 API、SMS-Man 等公开资料的归纳，SIM 接码平台通常具有以下能力：

### 2.1 用户侧通用能力
- 账户注册 / 登录
- 余额充值与账单
- 查询项目价格与库存
- 按 **国家 / 运营商 / 项目** 购买一个接码资源
- 轮询订单状态
- 获取验证码全文 / 完整消息
- 取消订单
- 标记号码不可用 / 被占用 / 被封
- 请求再次接码或继续等待
- 查询历史订单与成功率

### 2.2 典型订单状态
不同平台命名略有差异，但普遍都围绕以下状态运转：

- `PENDING`：资源准备中
- `ALLOCATED`：已分配资源，等待接收验证码
- `RECEIVED` / `STATUS_OK`：已收到验证码 / 消息
- `RETRY` / `WAIT_RESEND`：等待再次触发或再次接收
- `CANCELED`：用户主动取消
- `TIMEOUT`：超时未收到
- `FINISHED`：订单完成
- `BANNED` / `BAD_RESOURCE`：资源质量问题，被判定为不可继续使用

### 2.3 定价与库存逻辑
- 同一项目在不同国家 / 运营商价格不同
- 不同资源池的成功率不同，价格也不同
- 库存实时变化
- 平台往往允许设置最低价、最高价、溢价、供应商加价
- 部分平台支持 **按次验证码订单** 与 **长租号码** 两类模式

### 2.4 风控与平台规则
根据公开规则页与 API 文档，平台通常会：

- 对频繁取消、频繁退款的账户做限制
- 对异常 API 调用频率做限流
- 对资源被滥用、封号、恶意注册等行为做封禁
- 对上游质量差的资源做隔离、降权、下架
- 用评分 / 信誉分影响可购买资源范围与优先级

这些机制都非常适合迁移到邮件接码平台。

## 3. Nexus-Mail 的角色模型

Nexus-Mail 采用三类核心角色：

### 3.1 用户（Buyer / Client）
用户是购买邮件接码服务的一方。

#### 用户能力
- 注册 / 登录 / API Key 管理
- 余额充值、账单查看、退款记录查看
- 浏览项目目录（如 Google / TikTok / Discord / Amazon 等）
- 查询价格、成功率、可用库存、推荐域名池
- 创建一次性邮件接码订单
- 创建长租邮箱订单
- 查询订单状态
- 获取提取结果：验证码、链接、完整邮件原文
- 配置回调地址（Webhook）
- 查看历史订单、成功率、平均到达时延

#### 用户视角的核心体验
和 SIM 接码平台一致：
1. 先看价格 / 库存
2. 下单获取一个邮箱地址
3. 把邮箱地址填入目标站点
4. 轮询或等待回调
5. 拿到验证码 / 链接
6. 成功则完成，失败则取消 / 超时 / 退款

---

### 3.2 供应商（Supplier）
供应商负责提供底层资源，不一定是“卖邮箱账号”，更常见是提供：

- 域名
- Catch-All 收信能力
- SMTP / MX 边缘节点
- 别名生成服务
- 特定项目的高成功率邮箱池
- 邮件提取规则模板

#### 供应商能力
- 接入自己的域名或子域名
- 配置接收方式：MX、自建 SMTP、Webhook 入站、邮件转发
- 上传 / 管理邮箱池、别名池、规则池
- 配置可售项目、支持国家 / 区域 / 标签
- 配置成本价、库存、并发上限、日配额
- 查看订单命中率、验证码成功率、退单率、投诉率
- 收取结算款、查看冻结金额

#### 供应商在平台中的定位
参考 SIM 接码行业，平台不应只做单一资源源头，而应支持：

- **平台自营资源**
- **外部供应商资源**
- **混合调度资源**

这样才能形成：
- 更稳的库存
- 差异化价格
- 不同项目的专属高质量池
- 供应商间的竞价与评分机制

---

### 3.3 管理员（Admin / Operator）
管理员负责平台治理与商业运营。

#### 管理员能力
- 用户管理：封禁、限流、等级、授信、手工调账
- 供应商管理：准入、审核、分组、结算、下架
- 项目管理：定义项目、关键词规则、计费规则
- 定价管理：基础价、浮动价、利润率、促销价
- 风控管理：黑名单、灰名单、发件人规则、关键词规则
- 财务管理：充值、退款、冻结、对账、结算
- 运营管理：成功率看板、库存看板、到达延迟、供应商质量排行
- 审计管理：订单轨迹、操作日志、消息存证、争议仲裁

## 4. 核心产品模式

Nexus-Mail 至少应支持两种主要模式，这与 SIM 接码平台的“激活码模式 + 长租模式”高度对应。

### 4.1 一次性邮件接码（Activation Mode）
对应 SIM 平台的单次接码订单。

#### 适合场景
- 注册验证码
- 登录 OTP
- 重置密码邮件
- 邀请码邮件
- Magic Link 登录

#### 用户流程
1. 用户指定项目（如 `discord`）
2. 可选指定域名池 / 国家标签 / 供应商 / 价格上限
3. 平台创建订单并分配一个邮箱地址
4. 用户将该邮箱用于目标平台验证
5. 入站邮件被平台捕获
6. 平台提取验证码 / 链接 / 原文
7. 订单进入成功、超时、取消等终态

#### 计费逻辑
- 下单时预冻结金额
- 收到符合规则的首封邮件后正式扣费
- 在超时时间内未收到有效邮件则退回冻结金额
- 若用户主动取消且未收到邮件，可按规则退款

---

### 4.2 长租邮箱（Rental Mode）
对应 SIM 平台的租号模式。

#### 适合场景
- 需要多次接收邮件的长期项目
- 多轮登录 / 风控验证
- 长期账号维护
- 邮件回执 / 通知持续接收

#### 特点
- 提供固定邮箱地址，租期可按小时 / 天 / 周 / 月
- 可接收多封邮件
- 可以设置仅允许特定发件人或项目
- 支持查看全部邮件历史
- 支持续租、提前关闭、自动续费

#### 计费逻辑
- 按租期计费
- 可设置“首 X 分钟无有效邮件可取消退款”窗口
- 可收取独占费 / 保底费 / 大附件费

## 5. 推荐的订单生命周期设计

这是整个系统最重要的部分，建议直接参考接码平台的状态机思维。

### 5.1 一次性订单状态

```text
CREATED
  -> ALLOCATING
  -> ALLOCATED
  -> WAITING_EMAIL
  -> EMAIL_RECEIVED
  -> PARSING
  -> READY
  -> FINISHED
```

异常分支：

```text
WAITING_EMAIL -> TIMEOUT
WAITING_EMAIL -> CANCELED
ALLOCATED     -> BAD_RESOURCE
PARSING       -> PARSE_FAILED
ANY NONFINAL  -> EXPIRED
```

### 5.2 推荐状态说明
- `CREATED`：订单已创建，尚未锁定资源
- `ALLOCATING`：正在从域名池 / 邮箱池 / 供应商池分配邮箱
- `ALLOCATED`：已成功分配邮箱地址
- `WAITING_EMAIL`：等待目标邮件到达
- `EMAIL_RECEIVED`：已接收入站邮件
- `PARSING`：解析验证码、链接、邮件内容
- `READY`：提取结果可供用户获取
- `FINISHED`：用户确认完成或系统自动完成
- `TIMEOUT`：在规定时间内未收到有效邮件
- `CANCELED`：用户主动取消
- `BAD_RESOURCE`：供应商资源被判定不可用
- `PARSE_FAILED`：收到邮件但未能提取有效结果
- `EXPIRED`：超出保留期，订单关闭

### 5.3 状态、计费与触发动作对应表

| 状态 | 含义 | 是否扣费 | 是否可退款 | 谁可触发后续动作 |
| --- | --- | --- | --- | --- |
| `CREATED` | 订单已创建，未锁定资源 | 否，仅冻结 | 是 | 系统 |
| `ALLOCATED` | 已分配邮箱地址 | 否，仅冻结 | 是 | 系统 / 用户取消 |
| `WAITING_EMAIL` | 等待目标邮件到达 | 否，仅冻结 | 是 | 系统 / 用户取消 / 用户 retry |
| `EMAIL_RECEIVED` | 已收到候选邮件 | 否 | 视规则 | 系统 |
| `PARSING` | 正在解析验证码 / 链接 | 否 | 视规则 | 系统 |
| `READY` | 已得到可交付结果，用户可读取 | 是，正式扣费 | 一般否 | 用户 finish / 系统自动 finish |
| `FINISHED` | 订单终态，交付完成 | 是 | 否 | 系统 / 用户 |
| `TIMEOUT` | 规定时间内无有效邮件 | 否 | 是 | 系统 |
| `CANCELED` | 用户主动取消 | 否 | 是（未收到有效邮件时） | 用户 |
| `BAD_RESOURCE` | 资源不可用或命中错误 | 否 | 是 | 系统 / 管理员 |
| `PARSE_FAILED` | 收到邮件但无法提取有效结果 | 可配置 | 可配置 | 系统 / 管理员 |
| `EXPIRED` | 超过保留期关闭 | 已结算单保持已结算 | 否 | 系统 |

### 5.4 finish 与 retry 的明确语义

为避免实现歧义，建议把几个动作严格定义：

- `finish`：用户确认本次订单已完成使用；若用户未主动 finish，系统可在结果交付后自动 finish
- `retry_wait`：继续等待下一封满足规则的邮件，不重新分配地址
- `retry_parse`：对已收到的原始邮件重新执行提取规则
- `retry_reallocate`：判定当前资源质量差，废弃当前地址并重新分配新地址
- `extend_timeout`：延长等待窗口，但不改变资源

其中 MVP 可只实现两种：
- `finish`
- `retry_wait`

### 5.5 长租订单状态
- `ACTIVE`：租期内可正常接收
- `PAST_DUE`：到期未续费，等待关闭
- `EXPIRED`：租期结束
- `CLOSE_REQUESTED`：用户主动申请关闭
- `CLOSED`：订单关闭
- `SUSPENDED`：因风控或供应商故障临时暂停

## 6. 项目（Project）与资源（Resource）模型

SIM 接码平台最大的运营经验之一，就是 **项目不是简单标签，而是价格、成功率、提取规则、库存偏好、风控策略的绑定单元**。

因此在 Nexus-Mail 中，建议把项目定义为：

### 6.1 Project
一个项目对应一个目标站点或业务类型，例如：
- `google`
- `telegram`
- `discord`
- `tiktok`
- `generic-otp`
- `password-reset`
- `magic-link`

#### 每个项目包含
- 项目标识 / 展示名
- 支持的一次性模式 / 长租模式
- 默认超时时间
- 默认价格
- 支持的域名池
- 允许的发件人名单 / 黑名单
- 邮件主题关键字规则
- 正文提取规则
- 链接提取规则
- 成功判定规则
- 再次取码规则

### 6.2 Resource
资源不只是邮箱本身，而是一个可售卖的“接收能力单元”。

#### 资源维度可包括
- `domain`
- `mailbox`
- `alias`
- `supplier`
- `pool`
- `region`
- `quality_score`
- `success_rate`
- `cost`
- `concurrency_limit`
- `daily_limit`
- `status`

## 7. 邮件接码与短信接码的关键差异

虽然商业逻辑相似，但邮件比短信多出几个需要额外处理的点：

### 7.1 邮件内容结构更复杂
邮件可能包含：
- 主题
- 纯文本正文
- HTML 正文
- 多段 MIME
- 附件
- 图片验证码
- Magic Link

因此系统不能只做 4~6 位数字提取，而应支持：
- 数字验证码提取
- 字母数字 OTP 提取
- URL / Magic Link 提取
- 多规则回退提取
- 完整原文返回

### 7.2 同一邮箱可能收到多封不同类型邮件
必须有：
- 发件人过滤
- 时间窗口过滤
- 项目级匹配规则
- 邮件去重
- 首封有效邮件 / 最新有效邮件策略

### 7.3 邮件链路更容易受信誉和投递影响
因此需要监控：
- MX / SMTP 可用性
- SPF / DKIM / DMARC
- 到达延迟
- 退信率
- 被目标平台拒收的比例

### 7.4 资源形态更灵活
邮件平台可售卖的其实不是“账户”，而是：
- 一个一次性别名地址
- 一个固定邮箱地址
- 一个域名下无限别名能力
- 一个项目专用域名池

这让平台比 SIM 接码更容易做成多供应商、弹性库存和低边际成本模式。

## 8. 推荐的供应商接入模式

### 8.1 自营资源
平台自己维护：
- 域名
- MX
- SMTP 接收边缘
- 解析规则

优点：
- 质量可控
- 毛利高
- 数据完整

### 8.2 第三方供应商资源
供应商通过后台或 API 提供：
- 可售域名池
- 可售邮箱池
- 成本价
- 可用库存
- 项目白名单
- 回调地址或转发目标

优点：
- 扩库存快
- 覆盖项目广
- 降低单点风险

### 8.3 供应商归责与结算建议
为避免平台、用户、供应商之间发生扯皮，建议尽早明确归责规则：

| 场景 | 用户是否退款 | 供应商是否结算 | 说明 |
| --- | --- | --- | --- |
| 未收到任何邮件且超时 | 是 | 否 | 典型超时单 |
| 收到邮件但命中错误项目 / 发件人 | 是 | 否 | 视为坏资源或路由错误 |
| 收到邮件但解析规则失败 | 可配置 | 可配置 | 若原文可交付，可按“交付成功但提取失败”处理 |
| 收到有效验证码 / 链接 | 否 | 是 | 成功单 |
| 用户主动取消且未收到有效邮件 | 是 | 否 | 取消单 |
| 供应商转发明显延迟超 SLA | 通常是 | 通常否 | 可额外扣分 |
| 资源被判 `BAD_RESOURCE` | 是 | 否 | 同时记供应商质量扣分 |

建议供应商结算至少区分：
- 成功单
- 超时单
- 坏资源单
- 可争议单

并为每个供应商维护：
- 成功率
- 平均到达时延
- 争议率
- 退款率
- SLA 违约次数

### 8.4 混合调度
平台根据：
- 成本
- 成功率
- 延迟
- 供应商信誉
- 当前库存

自动选择最优资源。

#### 推荐调度策略
`综合得分 = 毛利权重 + 成功率权重 + 延迟权重 + 供应商质量权重 - 风险惩罚`

## 9. 平台需要具备的后台能力

### 9.1 用户后台
- 余额 / 账单
- API Key
- IP 白名单
- Webhook 管理
- 历史订单
- 失败原因查看
- 退款记录
- 用量统计

### 9.2 供应商后台
- 域名接入管理
- 邮箱池 / 别名池管理
- 库存与成本设置
- 订单明细与命中情况
- 供应商收益报表
- 退款 / 争议查看
- SLA 指标

### 9.3 管理后台
- 用户、供应商、项目三大控制台
- 实时库存图
- 实时订单流
- 成功率排行
- 退款率排行
- 风控事件中心
- 财务对账中心
- 审计日志

## 10. 计费、退款与信誉分机制

这是从 SIM 接码平台借鉴最有价值的部分之一。

### 10.1 余额模型
- 用户余额
- 冻结余额
- 可提现余额（供应商）
- 待结算余额（供应商）
- 平台收入

### 10.2 一次性订单计费建议
- 下单：冻结金额
- 成功收到有效邮件：扣费
- 用户取消且未收到有效邮件：退款
- 超时未收到：自动退款
- 资源异常：自动退款并给供应商记质量扣分

### 10.3 长租订单计费建议
- 下单即扣租金
- 在退款保护期内未收到任何有效邮件可部分或全额退款
- 超过保护期按使用成立
- 可支持续租折扣与批量租赁折扣

### 10.4 信誉分 / 风控分
对用户可建立评分模型：
- 高频取消率
- 高频超时率
- API 请求异常率
- 投诉 / 争议率
- 支付风险等级

对供应商也建立评分模型：
- 可用率
- 成功率
- 平均到达时延
- 退款率
- 被投诉率
- 被判 BAD_RESOURCE 比例

评分影响：
- 用户购买限额
- 供应商流量分配权重
- 价格浮动
- 结算周期

## 11. 风控与合规

Nexus-Mail 不应该只是“能收邮件”，而必须默认具备平台级治理能力。

### 11.1 风控能力
- API 限流
- IP 黑白名单
- User-Agent / 指纹分析
- 异常取消拦截
- 邮件爆量告警
- 发件人黑名单
- 关键词黑名单
- 项目级限购
- 单用户并发阈值

### 11.2 合规与平台边界
建议在 README 与产品规则中明确：
- 平台仅面向合法测试、自动化运维、账号安全验证、企业流程集成等用途
- 禁止用于垃圾注册、诈骗、撞库、薅补贴、恶意绕过平台规则等非法用途
- 平台保留封禁用户、冻结订单、拒绝结算、上报异常行为的权利
- 邮件保留期、日志保留期、隐私策略与申诉机制需明确

### 11.3 邮件原文与隐私权限建议
邮件接码平台比短信接码平台更容易涉及敏感正文，因此建议一开始就明确：

- 默认仅订单所有者可查看原文
- 管理员查看原文应有审计日志
- 供应商默认不应直接查看用户邮件原文，只看投递与状态元数据
- Webhook 默认应返回结构化提取结果，完整原文需单独权限开启
- 附件默认禁用或隔离存储
- 原文保留时间应可配置，例如 1~7 天
- 日志与导出中应对邮箱地址、发件人、链接参数进行脱敏

- 返回更接近传统接码平台的状态字段

这样可以降低迁移成本，同时保留平台内部更规范的 REST 模型。

## 16. 为什么这个方向成立

从商业与系统设计上看，邮件接码平台具备和 SIM 接码平台相同的几个成立条件：

- 存在明确、重复、可自动化的验证码接收需求
- 用户对“按次购买 + API 自动化”有天然需求
- 资源可以池化、标准化和多供应商化
- 成功率、延迟、库存、价格都可以量化运营
- 平台可以通过调度、定价和风控获得稳定利润

而相比短信接码，邮件接码还有额外优势：
- 域名与别名资源更灵活
- 理论库存弹性更大
- 单资源边际成本通常更低
- 更容易做精细化解析与内容结构化
